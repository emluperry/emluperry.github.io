PMDBot

JAVASCRIPT & DISCORD || PERSONAL

A Discord Bot Game inspired by Pokemon Mystery Dungeon

A very different project to others on my portfolio. This project was my second foray into developing for discord, and my first using javascript.

This project started as a glorified calculator for a Pokemon Mystery Dungeon themed tabletop roleplay, hosted over Discord. It was originally intended to be used to generate random encounters and calculate the effects of attacks, but the project grew into something much larger than I'd originally expected, spanning multiple concepts as time went on, as I had originally planned to do all of the calculations by hand. The first concept is from back in November, when I was still considering using standard playing cards to determine the random events!

Once I realised that I could easily achieve everything I wanted in the time given, I ended up skewing to a system that more closely resembled calculations present in the main series of games, repurposing speed to influence the order of attacks. This came with its own host of problems as well, since I would be using set levels for my players Pokemon and hadn't had the chance to fully playtest enemies and their movesets, which meant encounters could be far too easy or surprisingly difficult.

An example of what going into a dungeon is like.

To keep things simple, and because it fit for the context, each player would have three different Pokemon slots at level 10, 30 and 50, ith each generally representing a different evolution for their Pokemon where possible. Each of these Pokemon had predetermined movesets as well, since implementing experience and moveset choice was far beyond what I'd set out to do. At certain points in the campaign, I could cause someone's Pokemon to level up or evolve.

The nature of things meant there were actually a lot of people involved, that might have wanted to do things while I wasn't around or awake. I'm pretty proud of my solution for this, which was to make dungeon exploration purely user driven, provided all works well. Players could type a discord command into a given dungeon channel, tag in the other players that would be joining them on an exploration, and the bot would start randomly generating 'rooms' in the form of cards, similar to the original playing card concept. Each 'card' had its own chance of showing up, and could show up multiple times.

Cards would determine if the players ran into wild Pokemon, items, trainers, a shop, a monster house, or a way to advance, all of which would be controlled using buttons provided by the discord API. As a secondary failsafe, I knew I wouldn't be able to test for every situation before my deadline, and errors such as null references would shut down the entire bot. So I also made it so that the bot would track a group's progress in a dungeon, saving the participants, their status, and their progress within a dungeon. The only thing that was not tracked were items, as these were a global bag for anyone to use, and I did not intend on putting more work into the inventory with the scope of the project as it was.

Battles, random encounters or boss encounters setup by myself, would run in their own thread within discord, as it would generate a lot of messages. This was necessary to get certain information across, and to allow different players to choose between attacking, using items, or passing their turn. Rather ambitiously, my original intention for 'passing' a turn would be to allow for actions outside of the strict battle system, but this didn't quite come to fruition, and so the option was rarely used! Every turn, the bot would ping the associated player's discord tag and display the menu, and the player would pick an action, which the bot would then calculate. The bot could account for everything from damage, stats, type effectiveness, status effects, abilities, stat boosts, and weather or field conditions.

Being knocked out of a battle during a dungeon would knock that player out of the exploration if other players don't have or choose not to give them a reviver seed. Outside of dungeons, the players would also be able to browse items at the kecleon shop, which had a set of different items for each day of the session, predetermined by myself.

A lot of time for this bot went into getting the special effects of moves and abilities to work. One of the difficulties with developing this sort of system in javascript of all languages is the lack of classes, and by the time I realised that I could start looking into Typescript, I was running out of time to develop the bot and had already found a way around things by simply implementing repeated functions across various abilities as if I were just overriding certain base functions.
Using Sequelize as my database of choice, I ended up porting data for around 214 moves, 86 abilities, 198 different instances of various Pokemon, including wild Pokemon for 10 different dungeons. I also implemented 24 items in the end, mainly berries and status causing seeds.

There's a lot I could've done to improve this project, in hindsight, but as a project developed over the course of 8 months, alongside other coursework and responsibilities (such as planning the campaign!) I'm pretty happy with how it turned out. In an ideal world, I'd implement random shop stocks, requests, add experience, levelling up, and held items, and maybe implement a better way of setting up various items and dungeons, while also improving the codebase to be a little more maintainable. I don't intend on working on this project further, as I don't have any real need to, but it was great fun, and I enjoyed testing the limits of what I could do. (If I could figure out why the buttons would occasionally stop working for myself and my players seemingly at random, I'd love to know that too!)
